# Per-User Auth + RBAC — Evidence (2026-07-05)

WI-5 (Phase 2): per-user authentication (Auth.js / NextAuth v5 credentials
provider, bcrypt) with role-based access control enforced server-side on
`POST /api/commands` and audit attribution. Verified live + hermetically.

## Setup

- Migration `010_dashboard_users.sql` (additive: `dashboard_users` table +
  nullable `user_id`/`username` on `control_command_log`) — applied live.
- Two users created via `scripts/create-dashboard-user.mjs`
  (`op_test`/operator, `view_test`/viewer) for verification, then removed.

## Live verification (real Auth.js sign-in → command)

Full credentials flow: GET `/api/auth/csrf` → POST
`/api/auth/callback/credentials` (sets the session cookie) → POST
`/api/commands` (Basic tunnel gate + session cookie):

```
user=view_test  session_role=viewer    ->  POST /api/commands  HTTP 403
user=op_test    session_role=operator  ->  POST /api/commands  HTTP 200
audit row: username=op_test | command=OPEN_DOOR | decision=REJECTED
```

- **Viewer blocked with 403 BEFORE the safety gate** — zero Ditto writes.
- **Operator reaches the gate (200).** The subsequent `REJECTED` is the
  deterministic gate acting on twin state *after* RBAC admitted the operator —
  the correct layering (RBAC controls access to the route; the gate keeps
  authority over admission).
- **Audit attribution**: `control_command_log.username` is stamped with the
  authenticated user.

## Hermetic verification (Vitest, mocked auth())

`apps/dashboard/app/api/commands/route.test.js` (35 assertions total):

- `RBAC: a viewer session is blocked with 403 BEFORE the gate (zero Ditto writes)`
- `RBAC: an operator session may issue commands and is attributed in the audit row`
  (asserts the `control_command_log` INSERT params include the username).

## Design notes

- Auth.js session is the per-user identity; the existing HTTP Basic-Auth
  middleware stays as the OUTER demo-tunnel gate. `/api/commands` resolves the
  session first (`resolvePrincipal`), falling back to the Basic/trusted-local
  principal for API clients, tests, and local dev.
- Roles: `viewer` (read-only) < `operator` (commands) < `maintainer`
  (+access-control mutations) < `admin`. `src/server/authRoles.js` holds the
  pure RBAC decisions.
- The dashboard shows a read-only banner for viewer sessions; the server is the
  actual enforcement boundary.
- n8n user management is enabled (`N8N_USER_MANAGEMENT_DISABLED=false`); the
  owner account is set up once in the n8n UI (maintainer step).
