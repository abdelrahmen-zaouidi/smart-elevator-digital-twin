# Security baseline — Smart Elevator Twin

Scope of this baseline (the "necessary" encryption/auth for a local thesis rig):

1. **MQTT‑over‑TLS** on the ESP32 ↔ Mosquitto hop (the only link that crosses WiFi).
2. **Broker authentication + ACLs** — no anonymous clients; the elevator command
   topic is publish‑restricted.
3. **Secrets hygiene** — credentials/keys out of source, templates committed.

## What is protected (and what is deliberately not)

| Link / asset | Control | Rationale |
|---|---|---|
| ESP32 ↔ broker (over WiFi) | **TLS (8883), server‑only, ESP32 pins the CA** + username/password + ACL | only hop on an untrusted network |
| bridge / n8n ↔ broker (Docker net) | username/password + ACL, plaintext 1883 | intra‑Docker, never on the wire |
| dashboard browser ↔ broker (WS 9001) | username/password + ACL (read‑only identity) | same host; commands go via Ditto |
| All broker clients | **anonymous disabled** | closes unauthenticated command injection |
| Secrets at rest | gitignored `.env`, `apps/dashboard/.env.local`, `secrets.h`, `infra/mqtt/passwordfile`, `infra/mqtt/certs/` | keep creds/keys out of source |
| **Not done (by decision)** | Ditto HTTPS, dashboard HTTPS, intra‑Docker TLS, Mongo/Postgres at‑rest, mutual TLS, ESP32 flash‑encryption | localhost/Docker‑internal or production‑grade overkill for this prototype |

## Broker identities and ACL

Defined in `infra/mqtt/aclfile`, hashed in `infra/mqtt/passwordfile`.

| User | Publish | Subscribe |
|---|---|---|
| `esp32-elevator` | telemetry, events, status (own device) | commands (own device) |
| `bridge` | `elevator/+/commands` | telemetry, events, status |
| `agents` (n8n) | `elevator/+/commands` | telemetry, events, status, commands |
| `dashboard` (browser) | — (read‑only) | telemetry, events, status |
| `healthcheck` | `healthcheck/mqtt` only | — |

**Safety property:** only `bridge` and `agents` may publish to `.../commands`. The
ESP32 and the browser cannot command the elevator over MQTT (verified below).
The dashboard `/api/commands` route writes accepted operator intent to Ditto
only. It also stores a durable `features/control/properties/pending_command`
object with a unique `command_id`; bridge reconciles that intent and performs
the MQTT command fanout as the `bridge` broker identity. If the dashboard shows
an MQTT "Not authorized" failure while moving the cabin, restart `npm run dev`
so the route reloads the current Ditto-only command path.

## TLS

- Local CA + broker server cert generated by `scripts/gen-mqtt-certs.sh`
  (ECDSA P‑256). Server cert **SAN must include the IP the ESP32 connects to**
  (`192.168.100.7`); re‑run the script if that changes.
- Server‑only TLS (`require_certificate false`): the broker proves identity, the
  ESP32 authenticates with username/password. The CA (`ca.crt`, public) is pinned
  in firmware (`MQTT_CA_CERT` in the `.ino`, overridable via `secrets.h`).
- The ESP32 syncs time via NTP before connecting (TLS validity check needs a clock).
- Firmware toggle `MQTT_USE_TLS` (default `1`): set to `0` for a plaintext 1883
  auth‑only fallback if TLS misbehaves on the bench.

Reproduce against the live broker after cutover (host needs no MQTT client — use
the broker image):

```bash
# anonymous must fail:
docker exec elevator-mqtt mosquitto_pub -h localhost -p 1883 -t t -m x ; echo $?   # non-zero
# TLS from the ESP subnet (replace IP); validates the CA:
mosquitto_pub -h 192.168.100.7 -p 8883 --cafile infra/mqtt/certs/ca.crt \
  -u esp32-elevator -P "$MQTT_ESP32_PASSWORD" \
  -t elevator/building-floor1-elevator/telemetry -m '{"t":1}'
# ACL: esp32 must NOT be able to publish a command:
mosquitto_pub -h 192.168.100.7 -p 8883 --cafile infra/mqtt/certs/ca.crt \
  -u esp32-elevator -P "$MQTT_ESP32_PASSWORD" \
  -t elevator/building-floor1-elevator/commands -m x   # broker drops it
```

## Cutover (the one disruptive step — run when ready)

Changing the broker to require auth will disconnect every client until it has
credentials. Order:

1. Confirm secrets exist (already created here, gitignored):
   - root `.env` → `MQTT_*` passwords
   - `apps/dashboard/.env.local` → `NEXT_PUBLIC_MQTT_USERNAME/PASSWORD`
   - `firmware/main_esp_32_code_smart_elevator_v6/secrets.h` → ESP32 creds
2. Recreate the broker and the Docker clients:
   ```bash
   # --build matters: the bridge image must contain the MQTT auth code.
   docker compose up -d --build --force-recreate mosquitto bridge n8n
   docker logs elevator-mqtt --tail 20      # expect 1883/9001/8883 open, healthy
   ```
3. **n8n**: set the MQTT node credentials in the n8n UI to user `agents` /
   its password (n8n stores node credentials in its own encrypted store, not env).
4. **Dashboard**: restart `npm run dev` so it reloads `apps/dashboard/.env.local`.
5. **Firmware**: ensure `secrets.h` has the ESP32 creds, flash, and confirm the
   serial log shows NTP sync + `TLS enabled: port 8883, server CA pinned` and an
   MQTT connect.

Rollback: `MQTT_USE_TLS 0` (firmware) reverts the ESP32 to 1883 auth‑only; setting
`allow_anonymous true` in `elevator.conf` + recreate reverts the broker entirely.

## Credential rotation

```bash
# rotate one user (example: esp32-elevator)
docker run --rm -v "$(pwd -W)/infra/mqtt:/mc" eclipse-mosquitto:2 \
  mosquitto_passwd -b /mc/passwordfile esp32-elevator NEW_PASSWORD

# Docker Desktop bind mounts can preserve Linux file modes from the helper
# container. Mosquitto drops to uid 1883, so keep auth files readable only by
# that runtime user before recreating the broker.
docker run --rm -v "$(pwd -W)/infra/mqtt:/mc" --entrypoint chown eclipse-mosquitto:2 \
  1883:1883 /mc/passwordfile /mc/aclfile
docker run --rm -v "$(pwd -W)/infra/mqtt:/mc" --entrypoint chmod eclipse-mosquitto:2 \
  0600 /mc/passwordfile /mc/aclfile

# update the matching value in secrets.h / .env / apps/dashboard/.env.local, then:
docker compose up -d --force-recreate mosquitto
```

To rotate the **TLS CA/server cert**, re‑run `FORCE=1 bash scripts/gen-mqtt-certs.sh`,
update `MQTT_CA_CERT` in the firmware (or `secrets.h`), re‑flash, and recreate the broker.

## Git note

This project is now a **git repository published to GitHub** (Apache-2.0). The
committed `.gitignore` excludes `.env`, `apps/dashboard/.env.local`, `secrets.h`, the
broker `passwordfile`, and `infra/mqtt/certs/` — confirm no secret has ever
been committed before pushing (`git log -p -- .env secrets.h` and a secret scan).
Any credential that was ever committed or pushed (e.g. an old WiFi password)
must be **rotated**, because git history and remote mirrors persist deleted
content. Rotation procedures for broker users and the TLS CA are below.

## Bench checklist (firmware bits not testable without hardware)

- [ ] ESP32 completes NTP sync on your WiFi within ~8 s (`[TIME] ... ok`).
- [ ] ESP32 TLS connect to `192.168.100.7:8883` succeeds (CA validates; SAN matches the IP).
- [ ] Telemetry/status reach the dashboard; a dashboard command reaches the ESP32 (via Ditto → bridge → commands).
- [ ] RAM headroom OK with `WiFiClientSecure` (S3 + PSRAM).
- [ ] `MQTT_USE_TLS 0` fallback still connects on 1883 with auth.
- [ ] If the broker IP changes, regenerate certs (SAN) and re‑pin the CA.
